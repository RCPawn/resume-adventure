# 技术分享模块 — 设计文档

> 姊妹篇（实现与工程约定）：[`技术分享模块-技术文档.md`](./技术分享模块-技术文档.md)

## 1. 背景与目标

站点已具备「项目详情」等长文阅读能力，仓库 `docs/` 下积累了面向维护者与读者的技术说明（如接入说明、性能与统计修复记录）。本模块的目标是：**把这些文档以「技术分享 / 站内知识库」的形式呈现给访客**，与作品集叙事互补，而不是替代项目详情页。

**设计原则**

- **内容单一来源**：正文以 `docs/` 下 Markdown 为准，避免在 Vue 里再写一份长文。
- **阅读体验一致**：版式、代码高亮、暗色模式与现有长文页尽量对齐，降低认知成本。
- **渐进交付**：先支持列表 + 单篇阅读（MVP），搜索、标签聚合、RSS 等放在后续迭代。

## 2. 用户与典型场景

| 角色 | 场景 |
|------|------|
| 访客 | 从导航进入「技术分享」，浏览标题与摘要，点开感兴趣的一篇。 |
| 作者（你） | 新增或修改 `docs/` 下文章后构建部署，无需改业务路由表即可被收录（若采用构建期索引，见技术文档）。 |
| 回访者 | 通过收藏夹或站内链接直达某篇 `/share/:slug`。 |

## 3. 信息架构

### 3.1 路由（建议）

| 路径 | 说明 |
|------|------|
| `/share` | 技术分享首页：文章列表（标题、摘要、日期、可选标签）。 |
| `/share/:slug` | 单篇正文，`slug` 与文件名或索引中的 id 一致。 |

首页 `NavBar` 可增加「技术分享」入口（与 `#projects` 等锚点并列，或放在控制区），指向 `/share`。

### 3.2 内容与 `docs/` 的关系

- **纳入范围**：面向读者的技术说明、接入文档、架构/性能笔记等 Markdown。
- **暂不纳入**：若未来出现仅 CI/脚本使用的内部说明，可用 frontmatter `published: false` 或单独目录排除（见技术文档）。
- **现有文档**：如 [`不蒜子接入说明.md`](./不蒜子接入说明.md)、[`性能与统计修复说明.md`](./性能与统计修复说明.md) 可作为首发文章；是否展示由索引与发布字段控制。

### 3.3 列表页信息元素（MVP）

每条条目建议包含：**标题**、**摘要（1～2 行）**、**发布或更新日期**、可选 **标签**。排序默认按日期降序。

### 3.4 详情页信息元素

- 顶部：**标题**、日期、可选标签；可选「返回列表」。
- 正文：渲染后的 Markdown（与项目详情类似的阅读宽度与排版层级）。
- 页脚：可选「上一篇 / 下一篇」或返回列表（二期亦可）。

## 4. 体验与视觉

- **布局**：桌面端可采用「主栏正文 + 可选目录（TOC）」与 `ProjectDetail` 类似策略；移动端单栏、TOC 折叠或吸底。
- **主题**：继承站点亮/暗与 `github-markdown-css` 风格，避免单独一套文档皮肤。
- **代码与图表**：代码块高亮、Mermaid 与现有详情页行为一致（含加载失败时的友好提示）。
- **国际化**：设计预留中英文切换；MVP 可仅展示中文 `docs`，英文通过独立文件或后续 i18n 扩展（技术文档写实现选项）。

## 5. 非功能需求（设计视角）

- **可维护性**：新增文章以加文件 + 更新索引为主，流程简单。
- **可发现性**：站点地图或 `robots` 是否包含 `/share` 由你决定；设计层建议单篇可被搜索引擎抓取（若整站公开）。
- **统计**：若沿用不蒜子，需明确「按页 PV」与列表页、详情页的计数语义，避免与现有说明冲突（参考 [`不蒜子接入说明.md`](./不蒜子接入说明.md)）。

## 6. 里程碑

| 阶段 | 范围 |
|------|------|
| **MVP** | `/share` 列表、`/share/:slug` 详情、构建期索引、`docs` 内 Markdown + frontmatter。 |
| **二期** | 标签筛选、全文搜索（前端索引或轻量接口）、上一篇/下一篇、RSS。 |
| **三期** | 英文镜像、阅读进度、评论或外链 GitHub Discussions。 |

## 7. 开放决策（实现前确认）

1. **URL 前缀**：固定为 `/share` 还是更短如 `/notes`（影响导航文案与外链）。
2. **是否必须 frontmatter**：无元信息时的默认标题（是否用文件名）与默认日期（是否用 git 时间，技术文档说明取舍）。
3. **与「项目详情」边界**：项目页继续承载项目 README；技术分享只挂 `docs/` 中「跨项目」的通用文档，避免重复。
