# 不蒜子（Busuanzi）接入说明

本站未在 `index.html` 中挂载官方 `busuanzi.min.js`，而是在 **`src/utils/busuanziApi.js`** 中自行实现与官方脚本等价的 **POST 请求 + 按 id 写 DOM**，以便在 **Vue SPA 路由切换**后仍能更新统计，并与页脚展示、详情页「本页阅读量」共用同一套逻辑。

---

## 1. 核心模块：`requestBusuanziApply()`

### 1.1 请求行为

- **地址**：`https://cdn.busuanzi.cc/api.php`
- **方法**：`POST`，`Content-Type: application/json`
- **请求体**：`{ url, referrer }`，其中 `url` 取发起请求时的 `location.href`，`referrer` 取 `document.referrer`（无则为空字符串）。
- **含义**：与不蒜子 CDN 脚本行为一致；服务端按 **每次成功计数的一次 POST** 参与 PV/UV 等统计（具体规则以不蒜子为准）。

### 1.2 响应如何落到页面

响应需为 JSON。模块遍历其 **键名**，仅处理匹配正则 `^busuanzi_[a-z0-9_]+$`（忽略大小写）的字段，然后：

```text
document.getElementById(键名).innerText = 值
```

即：**页面必须预先存在对应 `id` 的节点**，官方返回的 key（如 `busuanzi_site_pv`）与 DOM `id` 一致时才会被写入。使用 **`innerText`** 而非 `innerHTML`，可降低服务端若返回异常片段时的脚本注入风险（仍依赖不蒜子返回可信数值字符串）。

### 1.3 并发与防抖（避免短时间重复计数）

SPA 中常在同一时刻触发多处调用（例如：Footer `nextTick`、路由刚稳定、详情页 `fetchMarkdown` 的 `finally` 里再 `nextTick`）。

模块策略：

| 机制 | 作用 |
|------|------|
| **Trailing debounce（约 420ms）** | 短时间多次调用合并为 **一次** 实际 POST，减少同一导航瞬间的重复计数。 |
| **`inFlight` 共用 Promise** | 若上一次请求尚未结束，再次调用 **直接返回进行中的 Promise**，不额外发起请求。 |
| **`AbortController`** | 发起新请求前中止上一轮未完成请求，避免旧响应在 URL 已变化后仍写 DOM。 |

写 DOM 前还会校验：`signal.aborted` 或当前 `location.href` 与发起请求时捕获的 `url` 不一致则 **丢弃结果**，避免切换路由后旧数据刷到新页面。

## 2. 页脚（Footer）：全站 PV / 当日 UV 等

**文件**：`src/components/FooterSection.vue`

### 2.1 与不蒜子直接相关的 DOM

模板中为不蒜子预留了 **带固定 id 的节点**（脚本写入纯数字等文本；展示文案由 Vue 再包一层 i18n）：

- `busuanzi_site_pv`：全站 PV（统计写入后，再由脚本同步到 Vue 维护的 `sitePvText`）
- `busuanzi_today_uv`：当日 UV（同上，`todayUvText`）

实际流程是：**`requestBusuanziApply()` 先按 id 把不蒜子返回写进上述 span**；Footer 在延迟（约 180ms + 1200ms）后 **读取 DOM 文本**，抽取数字，写入 **localStorage**（`footer_busuanzi_cache_v1`），并用 `t('footer.statsPvFormat', …)` 等形式更新界面，以便 **多语言格式化** 与 **失败时用缓存避免显示 0**。

### 2.2 何时触发 `refreshBusuanzi()`（内部会调 `requestBusuanziApply()`）

- 组件 **挂载**后（先 `hydrateFromCache` 再刷新）
- **`route.fullPath` 变化**（换路由 / 查询串变化）
- 文档 **从隐藏变为可见**（`visibilitychange === 'visible'`）
- **`pageshow`**（例如浏览器往返缓存恢复）

**不再使用**定时器周期性刷新（历史上每分钟一次会导致长时间停留重复计 PV；见 `docs/性能与统计修复说明.md`）。

切换 **语言 `locale`** 时仅根据缓存/当前状态 **重算文案**，不单独为多语言再打一遍统计请求。

## 3. 学习足迹详情页：本页阅读量

**路由**：`/project/:id`  
**文件**：`src/views/ProjectDetail.vue`

### 3.1 DOM 约定

顶栏展示「本页阅读量」处放置：

- `id="busuanzi_page_pv"` 的空节点（`v-once`，避免 Vue 后续覆盖不蒜子写入的文本）

不蒜子若返回带该 key 的字段，会由 **`busuanziApi.js` 统一写入**。

### 3.2 何时请求

在 **`fetchMarkdown` 的 `finally`** 中：`loading` 置为 `false` 后 **`await nextTick()`**，再调用 **`requestBusuanziApply()`**。

即：**每次完成一次 Markdown 加载流程**（含失败分支进入 `finally`）会参与合并窗口内的一次 POST；切换项目 `id`、重新拉取文档时会再次走到这里，与 Footer 在同一防抖窗口内 **合并为单次请求** 是预期行为。

详情页 **没有**单独的定时刷新；长时间停在同一文档不会因本站逻辑而额外周期性加计 PV。

---

## 4. 与其他页面的关系

当前仓库内 **唯一直接调用** `requestBusuanziApply` 的即为：

- `FooterSection.vue`（经 `refreshBusuanzi`）
- `ProjectDetail.vue`（`fetchMarkdown` → `finally`）

首页等其他路由若未挂载 Footer，则不会出现页脚统计节点；但只要挂载 Footer，上述触发条件仍成立。

---

## 5. 依赖与隐私提示

- 统计请求发往 **第三方域名**（不蒜子 CDN），会携带 **当前完整 URL 与 referrer**，属于常见站点统计行为；若有合规需求需在隐私政策中说明。
- 页脚数字的 **本地缓存**仅存 PV/UV 数字字符串，不含密钥。
