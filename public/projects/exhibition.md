# 🏛️ 非遗虚拟展陈与数据监测平台

> 基于 Vue 3 与 Spring Boot 的展陈、图谱与数据可视化一体化实践，面向文化遗产数字化保护与传播。

---

## 📖 项目概述

本项目将白族非遗资源沉淀为可浏览、可检索、可统计的数字载体：前台侧重 **3D 展陈、传承图谱、影像与声音档案** 的沉浸式体验；后台支撑 **分类、展品、审核与内容运营**；监测层通过 **数字驾驶舱** 聚合指标，并对大屏依赖接口完成 **压测与量化**，便于上线与扩容决策。

### 能力一览

| 维度 | 说明 |
|------|------|
| 🎨 **3D 展陈** | 基于 Google Model Viewer，展品可旋转、缩放，适配多端 |
| 📜 **传承图谱** | 翻书式入口 + 力导向关系网络，呈现师徒与层级结构 |
| 📸 **纸上乾坤** | 瀑布流图集与沉浸式看图，兼顾检索与阅览节奏 |
| 🎵 **三道余音** | 音频档案与说明文案同屏，强化非遗听觉维度 |
| 📊 **数据驾驶舱** | ECharts 多维视图与热点趋势，服务运营与展示大屏 |
| 🌐 **门户与互动** | 文化门户导航、点赞收藏与评论等轻互动 |

## 🏗️ 系统架构

系统按 **终端 → 前端 SPA → HTTP API → 领域服务与持久化 → 缓存与静态资源** 分层拆分职责：公众浏览与管理运营共用同一后端，通过 **JWT + 权限模型** 区分路由；驾驶舱依赖的 **统计聚合接口** 走缓存热点路径，与 OLTP 写路径解耦；体积较大的 **GLB / 影像 / 音频** 走静态资源通道，减轻应用服务器字节流压力。

### 逻辑分层（前后端分离）

```mermaid
flowchart TB
  subgraph client["终端"]
    U["浏览器 / WebView"]
  end

  subgraph spa["前端 · Vue3 & Vite"]
    direction LR
    FE_P["公众端<br/>门户 · 展厅 · 图集 · 音频"]
    FE_D["驾驶舱<br/>ECharts"]
    FE_A["管理端<br/>分类 · 展品 · 审核"]
  end

  subgraph api["后端 · Spring Boot"]
    direction TB
    GW["REST Controller"]
    AUTH["JWT<br/>鉴权 / 拦截"]
    SVC["业务域<br/>展品 · 图谱 · 审核 · 统计"]
    DAO["MyBatis<br/>持久层"]
  end

  subgraph store["数据与资源"]
    direction LR
    MYSQL[("MySQL · 档案")]
    REDIS[("Redis · 缓存")]
    STATIC["静态资源<br/>GLB / 媒体"]
  end

  U --> spa
  FE_P -->|"HTTPS / JSON"| GW
  FE_D -->|"HTTPS / JSON"| GW
  FE_A -->|"HTTPS / JSON"| GW
  GW --> AUTH --> SVC --> DAO --> MYSQL
  SVC <--> REDIS
  spa -->|"静态 URL"| STATIC
```

**读图要点**：

- **横向**：同一 SPA 内划分公众、驾驶舱、管理端三类界面入口，共享 Axios 与路由守卫，但 **API 命名空间与权限要求不同**（匿名可读展品、写操作与审核需登录角色）。
- **纵向**：Controller 只做参数校验与编排；统计类接口在 Service 内聚合多表或预聚合结果，配合 **Redis** 降低驾驶舱轮询压力；大屏相关接口的容量边界见后文压测章节。
- **静态资源**：Model Viewer 加载的 GLB、瀑布流大图等优先走 **CDN / Nginx 静态目录**（图中统称静态资源），与 JSON API 分离，便于独立扩容与缓存策略。

### 驾驶舱数据流（简要）

驾驶舱刷新本质是「前端定时或触发拉取 → 统计接口 → 缓存命中则直接返回 → 未命中则读库聚合后回填缓存」。若需更强实时性，可在后续迭代引入消息队列或变更订阅，当前文档中的实现以 **可压测、可预期的 REST + Redis** 为主。

```mermaid
sequenceDiagram
  participant V as 驾驶舱前端
  participant B as 统计 API
  participant R as Redis
  participant M as MySQL

  V->>B: GET 大屏聚合指标
  alt 缓存命中
    B->>R: 读取统计快照
    R-->>B: 返回
  else 未命中
    B->>M: 聚合查询
    M-->>B: 结果集
    B->>R: 写入 TTL
  end
  B-->>V: JSON 指标
```

## ⚙️ 大屏接口压测说明

数字驾驶舱的卡片与图表依赖统计聚合接口（例如大屏场景的 `GET /heritage/stats/dashboard` 一类接口）。在集中参观或开幕式投影等情形下，接口需在有限硬件上保持稳定吞吐与低错误率。使用 **Apache JMeter 5.6.3** 配置线程组（如 **50 并发**、**5 秒** 爬坡）、HTTP 取样器与聚合报告，在百万级采样量级下观测到平均响应约 **6 ms**、吞吐量约 **7800+ 请求/秒**、**错误率 0%**，作为性能侧的支撑材料而非替代线上监控。

![JMeter：线程组、HTTP 请求与聚合报告](exhibition.assets/大屏接口压测.png)

## 🎨 设计与体验要点

1. **视觉基调**：以黑白灰为主、渐变点缀，贴合「尚白」审美倾向，避免抢眼配色干扰文物主体。
2. **动效节制**：点赞、收藏、翻书等动画保持可感知且短促，尊重 `prefers-reduced-motion` 场景。
3. **响应式**：门户与详情在笔记本高度与 27″ 大屏上均需限制主内容 `max-height` 与留白，避免一屏只剩半张图或按钮溢出（实现细则见项目内各页 `scoped` 样式约定）。
4. **性能**：GLB 压缩、图片懒加载、热点接口 Redis 缓存与数据库索引协同。
